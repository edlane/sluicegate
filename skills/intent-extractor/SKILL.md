# Cleanroom Intent Extractor Skill

This skill provides a highly disciplined, standard operating procedure for extracting system architecture, component intents, data structures, and systemic capabilities from an existing repository (`../ebb-agent`) and saving them as cleanroom reference markdown files in this repository (`/home/lane/develop/sluicegate/reference/`).

## Core Directives

> [!CAUTION]
> **Strict Cleanroom Rule: ZERO Source Code Copying**
> * Do **not** copy-paste or duplicate any implementation code from the source repository.
> * Do **not** write raw python, C, or bash code snippets from the source repo into the reference documents.

> [!IMPORTANT]
> **Decoupling Rule: ZERO Naming or Data Structure Replication**
> * Do **not** replicate the actual class names, function names, variable names, file structures, or specific naming schemes found in the source code.
> * Do **not** replicate the exact physical layout of JSON fields, database schemas, or filesystem extended attribute keys.
> * Instead, **decouple the spec from the implementation**:
>   * Create generic, clear, logical names for components, processes, classes, and methods (e.g. rename a class `FlowFile` to `AbstractSequentialLogStream` or similar).
>   * Describe all JSON schemas and extended attribute maps conceptually, mapping their semantic purpose rather than hardcoded keys (e.g., instead of documenting the literal key `"user.EBB.MAX_RETENTION_BLOCKS"`, document it as the "Block Retention Capacity Threshold attribute").
>   * Justify and describe the functional intent sufficiently so that a cleanroom developer can design a fully working, identical-behavior system from scratch without duplicating any of the original code's names or structures.

> [!TIP]
> **Allowed Artifacts**
> * High-level language-agnostic pseudocode or flowcharts (e.g. Mermaid diagrams) representing algorithms are fully allowed and encouraged.
> * Abstract class/function signatures showing logical inputs and outputs (using generic, decoupled naming) are allowed and encouraged.

---

## Step-by-Step Procedure

1. **Target Identification:** Identify the target file, directory, or subsystem in `../ebb-agent` to be analyzed.
2. **Path Setup:** Determine the output reference path in `sluicegate/reference/`. The reference structure should match a logical, cleanroom structure (e.g. `src/flowfile.py` becomes `reference/sequential_storage.md`).
3. **Deep Structural Analysis:**
   * Inspect the target file using safe research/viewing tools.
   * Understand the functional capabilities, data flows, state changes, file I/O operations, and interactions with other modules.
   * Conceptualize all classes, functions, and data structures. Translate them into descriptive, abstract concepts, renaming all variables, attributes, keys, and methods to a fresh, clean naming convention.
   * Formulate the key "intent" of the component—the core reason it exists.
4. **Draft the Reference Document:** Draft the markdown file using the **Abstract Decoupled Template** below.
5. **Verify Cleanroom Integrity:** Ensure no literal naming schemes, literal JSON keys, literal database fields, or raw source code blocks were replicated.
